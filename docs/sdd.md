# SDD Overview

## Purpose

このリポジトリは `SDD` で進める。

ここでいう `SDD` は、先にソフトウェアの意図と構造を明文化し、その設計を更新しながら実装を進めるやり方を指す。仕様、制約、アーキテクチャ、データフロー、運用上の注意をコードと同じ重みで管理する。

## Documents

- `docs/design.md`
  マスター設計書。日付付きで増やさず、このファイルを継続更新する。
- `docs/manual.md`
  利用者と導入担当者向けのマニュアル。Chrome への読み込み、使い方、制約、トラブル時の確認点をまとめる。
- `README.md`
  リポジトリの入口。短く全体像を伝え、詳細は上記文書へ誘導する。

## Rules

- 実装変更時は、必要に応じて `docs/design.md` も同時に更新する
- 利用方法や制約が変わる変更では `docs/manual.md` も更新する
- 日付付きの設計ファイルは増やさない
- 実装より先に、少なくとも制約と設計意図を文書で説明できる状態を保つ

## Current Design Stance

- 外部サーバー送信なし
- `GitHub.com` と `GitHub Enterprise` の両対応を前提にする
- 既存ブラウザセッションを使う
- Markdown rich diff 上のレビュー体験に集中する
- 参照元 `MIT` ライセンスの条件を維持する
